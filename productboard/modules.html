Onflo Modules — department modules, agent management & permission sets

What you're delivering

A clickable, design-mode prototype of the Onflo Modules feature set — three projects on one shared spine, built with the Onflo Design System. The UI is exact (every label, limit, and empty state is the intended behavior); the backend is intentionally not built.

  1. Department Modules — the module catalog, the request flow (prebuilt + custom), and the top-nav department switcher, at /settings/department-modules
  2. Agent Management — the agent directory, full-page agent profiles, create/edit forms, teams, and district 2FA, at /settings/agent-management
  3. Permission Sets — the capability-bundle catalog and full-area editor (a distinct project whose UI lives inside Agent Management)
  4. A persona-driven demo harness — swap the signed-in persona with ⌘K, switch module context, and trigger every error state on demand

Why this matters

A department module splits one district account into departments — Transportation, IT, HR — that share the system but not each other's work: tickets route to their own department, work can still hand off across departments, and integrations stay shared. Agent Management and Permission Sets then control who belongs to each module and what they can do there. The prototype exists so product, design, and engineering can agree on exactly how all of this behaves before anyone builds it for real.

Department Modules

The module catalog, how new modules are requested, and the switcher that moves you between them.

  • Catalog page groups modules into Active, Available, and Coming soon; the prebuilt set is Classic, Transportation, IT, HR, and Facilities (Classic and IT released today), plus custom modules (seed example: Music)
  • Request flow — prebuilt modules confirm-and-send; custom modules require a unique name plus an icon (~30 filled symbols) and color (20 presets) with a live preview; requests land in Active under an Under review overlay
  • Department switcher in the top nav — active modules only, a Global row for global admins, and each row labeled with its mapped permission set (Admin → Department Admin, Agent → Team Member)
  • Lifecycle — today: request → our team sets it up → Active; a reversible, lossless on/off/trial lifecycle is in design

Agent Management

One screen, reachable by any admin, that manages agents, teams, and district authentication — always scoped to the current switcher context.

  • Four tabs — Agents (always), Authentication (Global context only), Teams (department context only), Permission Sets; a context switch that invalidates the active tab falls back to Agents
  • Agents tab — a 12-column directory with status pills and a Permission Sets column resolved from each agent's per-module assignment; clicking a row opens the profile
  • Agent profile — a full page at /settings/agent-management/:id that opens in its own browser-style top-nav tab (several can be open at once), with Details, Permissions, and Activity tabs
  • Create/Edit Agent — one modal for both modes: personal info, repeatable per-module access rows, activation radios mapping to Pending/Active/Inactive, on-submit validation, and locked fields for synced agents
  • Teams — single-department teams; manual teams editable, synced teams (Active Directory/Azure/Google) read-only and owned by Integration Hub
  • Authentication — the district 2FA panel: a toggle plus a verification-method choice (Email / SMS / Both) once enabled

Permission Sets

Named capability bundles assigned per module to agents and to teams.

  • Catalog — 8 seed sets (6 System, including the locked Global Admin and Department Admin, plus 2 Custom); the table is context-scoped, and System sets open read-only
  • Editor — full-area with Details · Data Visibility · Actions · Settings tabs, per-section presets (No Access/View Only/Full Access/Custom), buffered assignment edits, and a docked save bar; global-tier sets show only Details + Settings
  • Capability catalog — 123 permissions (68 Actions + 55 Settings) whose Settings half mirrors the product's Settings sidebar 1:1
  • Single source of truth — one dataset read by eight surfaces (tables, profile, forms, switcher labels); the editor is the only writer

Technical documentation

Engineering Architecture — Onflo Modules Prototype — the single technical reference: how the three projects fit together, what's built vs mocked vs planned (✅/🟡/⛔), and how to drive every flow yourself.

Out of scope (don't touch)

  • Integration Hub itself — a separate team owns it; this prototype only demos the department-scoped access behavior (the Marketplace assignment UI is theirs and isn't built here)
  • The module on/off lifecycle — the goals are agreed (reversible and lossless) but the design is still open; today's flow stays request → setup → Active
  • Backend and persistence — everything is intentionally in-memory; the data tables are visual simulations marked TODO eng: replace with AG Grid + onfloTheme

DM-1

As a district admin I want the Department Modules page to group modules into Active, Available, and Coming soon So that I can see what we own, what we can add, and what's on the roadmap


Acceptance criteria

  • Active shows owned modules plus requested ones pending under an Under review overlay
  • Available shows released prebuilt modules not yet added, plus the Custom module request card
  • Coming soon shows unreleased prebuilt modules as muted previews with a Coming soon label and no request button
  • The prebuilt set is Classic, Transportation, IT, HR, and Facilities; each card reflects its contents (ticketing, analytics, workflows, and asset management vary by module)


DM-2

As a global admin I want to request an available module with a single confirmation So that my district can start using it without a support ticket


Acceptance criteria

  • The request opens a confirmation dialog; Send request shows Sending... then the success toast Request submitted for "[name]".
  • The module moves to Active under an Under review overlay
  • On failure the dialog stays open and shows Couldn't send your request. Please try again.


DM-3

As a global admin I want to request a custom module with a name, icon, and color So that departments without a prebuilt module get their own branded divider


Acceptance criteria

  • Name is required and unique (case-insensitive) across all modules; an empty or duplicate name shows an inline field error and blocks Send
  • Icon comes from a preset library of ~30 filled Material Symbols; Color from a preset palette of 20 colors
  • Both pickers drive a live card preview; once requested, the module renders like a prebuilt one (tinted tile + filled glyph)
  • Contents are fixed at ticketing + asset management — only the identity is chosen


DM-4

As an admin with more than one module I want a department switcher in the top nav So that I can move between my departments (or Global) without losing my place


Acceptance criteria

  • The switcher shows only for users with more than one module; global admins always see it; it lists active modules only
  • The trigger shows the current context (the module's icon + name in its color, or a globe + Global)
  • The menu adds a Global row for global admins; each module row is labeled with its mapped permission set (Admin → Department Admin, Agent → Team Member); the current context is checked
  • Selecting a module re-renders the whole app into that context; a failed switch shows the error toast Couldn't switch to [Module]. Please try again.


DM-5

As a district admin I want modules to turn on and off reversibly — including trials and removal So that changing our setup never risks data or shared resources


Acceptance criteria

  • Activation supports more than requests: support-driven, self-serve, and trial paths
  • Turning a module off, ending a trial, or removing it is lossless; re-adding restores the prior state
  • Shared resources (e.g. integrations) are never disturbed by a module change
  • Today's shipped flow is request → setup → Active; this flexible lifecycle is the agreed direction, with design in progress


AM-1

As an admin I want a searchable agent directory scoped to my current context So that I can find and manage the people in my department


Acceptance criteria

  • Columns: Name, Email, Permission Sets, Status, Module(s), Teams, Locations, Phone, Source, Job Title, Last Login, Date Added
  • Status renders as a pill: Active (green), Pending (yellow), Inactive (grey)
  • Permission Sets resolves each agent's per-module assignment to set names, de-duplicated and in module order
  • The Global context shows every agent; a department context shows only its own
  • Clicking a row opens that agent's profile; Create Agent opens the agent form


AM-2

As an admin I want a full-page agent profile that opens in its own top-nav tab So that I can keep several agents open and compare them


Acceptance criteria

  • The profile lives at /settings/agent-management/:id; each open profile is a closeable top-nav tab, and overflow collapses into a … tab
  • The hero card shows an XL avatar, the name, a status pill, {source} · {email}, and an Edit button
  • Tabs: Details (12 basic-info fields), Permissions (a card per module with its permission-set pill and Teams · {n}), Activity (Date, Time, Action, Description, with expand and CSV export)
  • A Pending agent gets a Resend activation email button → toast Activation email resent to {email}.
  • An unknown id shows Agent not found


AM-3

As an admin I want one create/edit agent form with per-module access rows So that setting up and maintaining an agent is a single flow


Acceptance criteria

  • Sections: Personal Information; Modules & Access as repeatable rows (Department Module required, Permission Set optional, Teams multi-select) with "Add another department module"; Custom Fields; Activation (create) / Account Status (edit)
  • Validation runs on submit: First Name, Last Name, Email (format-checked), and each row's Department Module; an invalid submit keeps the form open, surfaces the errors, and focuses the first invalid field
  • Activation radios map to status: Send activation email → Pending, Activate user → Active, Do not activate → Inactive
  • Synced agents show an info banner and lock the fields owned by the sync source
  • Success toasts: Agent created. / Agent updated.


AM-4

As a department admin I want teams that belong to my department and share a permission set So that groups of agents get consistent access in one step


Acceptance criteria

  • The Teams tab shows only in a department context; columns: Team Name, Agents, Permission Set, Source, Last Updated
  • Create Team has no Department field — a new team belongs to the department you're in
  • Manual teams open editable; synced teams (Active Directory, Azure, Google) open read-only with an info banner — they're edited in Integration Hub
  • Members are added via Add agents… (with search) and shown as chips; Team Name is required on submit
  • Success toasts: Team created. / Team updated.


AM-5

As a global admin I want district-wide two-factor authentication settings So that sign-in security is controlled in one place


Acceptance criteria

  • The Authentication tab appears only in the Global context (it's district-level)
  • The panel is titled Two Factor Authentication (2FA) with a toggle: 2FA Enabled / 2FA Disabled (default off)
  • Enabling reveals a Verification method radio group: Email / SMS / Both


AM-6

As a district admin I want each person's nav and Settings derived from their assignments So that people only see what they can actually use


Acceptance criteria

  • Assets shows only in a module that includes Asset management; Analytics only with Dashboard analytics and never for agents; Settings is hidden from agents entirely (direct navigation redirects away)
  • Settings sections gate by context + capability: Global is Global-context-only; General (which hosts Agent Management) shows for any admin; Call Center stays district-level
  • Agent Management's tabs adapt to context (Authentication Global-only, Teams department-only) and fall back to Agents when the active tab no longer applies


PS-1

As an admin I want a context-scoped catalog of permission sets So that I can see the capability bundles available where I am and who holds them


Acceptance criteria

  • Columns: Name, Type (System blue / Custom grey), Agents (assigned count, context-scoped), Last Updated
  • The Global context shows only global-tier sets (Global Admin); a department shows the system-wide sets plus its own custom sets
  • Clicking a row opens the editor; all six System sets open read-only
  • 8 seed sets: Global Admin, Department Admin, Global User, Team Member, Recorder, Read Only (System) plus IT Desk Lead and Classic Triage (Custom)


PS-2

As an admin I want to create a permission set from a name, description, and optional Copy From So that new roles start close to an existing one


Acceptance criteria

  • Create permission set opens a small modal: name (required), description, and an optional Copy From
  • A blank name blocks with Please fill in the required fields highlighted below. and the field helper Name is required.
  • There is no Department field — the current context applies implicitly, and a set created in the Global context is global-tier
  • On create the set is added to the catalog and its editor opens


PS-3

As an admin I want a full-area editor with Details, Data Visibility, Actions, and Settings So that one screen defines everything a set grants


Acceptance criteria

  • The editor opens full-area (replacing the tab bar) with a docked save bar — You have unsaved changes, Save Changes / Discard
  • Details — name, description, and assigned teams/agents as removable chips with a searchable Add Agents menu; assignment edits buffer and commit on Save or revert on Discard
  • Data Visibility — Tickets and Assets scopes (All / Assigned) plus an Assets filter builder (Asset Type / User Type / Grade, each with an Exclude option)
  • Actions & Settings — a section nav + search beside a permission grid; rows are toggles or Hide/View/Manage segments with sub-checkboxes under Manage; per-section presets No Access / View Only / Full Access / Custom
  • Global-tier sets show only Details + Settings and only the district-wide sections (Global, Integration Hub, Call Center); department sets show the department-tier sections (General, Automations, Tickets, Assets)
  • Save shows Permission set saved.; a blank name flips the save bar to an error — Fix the required fields before saving. — jumps to Details, and flags Name is required.


PS-4

As an admin I want every screen to read permission sets from one dataset So that names, counts, and assignments never disagree


Acceptance criteria

  • Eight surfaces read the same dataset: the agents-table Permission Sets column, the profile's Permissions tab, the agent form's options, the teams-table column, the team form's options, the editor, the editor's assigned list, and the switcher's role labels
  • Sets are referenced by id everywhere and resolved to names for display
  • The editor is the only writer, and there is no delete


PS-5

As a district admin I want every Settings nav item gated by a matching permission So that what someone can open is exactly what their set grants


Acceptance criteria

  • The Settings half of the permission catalog mirrors the Settings sidebar 1:1 — the same seven sections in the same order (Global · General · Integration Hub · Automations · Tickets · Assets · Call Center), one permission per nav item
  • The granted level gates the item: Hide → not shown, View → read-only, Manage/Edit → editable; a sub-group (e.g. Topic Manager) gates as a unit
  • The two lists stay in lockstep: adding or renaming a nav item adds or renames the matching permission
  • This is a spec for engineering — not enforced in the prototype yet
